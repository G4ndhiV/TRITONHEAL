\section{Objetivo experimental (1.1)}

\textbf{Problema.} Evaluar si un método de \textbf{generación de kernels Triton DSL} basado en SLM, con verificación y reparación pre-compilación, produce kernels compilables y un pipeline más eficiente que baselines comerciales o abiertos.

\textbf{Qué se demuestra.} (1) Tasa de compilación por configuración; (2) speedup proxy del pipeline en M4; (3) calidad del filtrado (MSR/FNR) en verificación del benchmark fijo.

\textbf{Método propuesto.} \textbf{Triton Heal dual}: generación con Llama-3.1-8B + reparación con DeepSeek-Coder-V2 (hasta 2 reintentos) + verificación/veto del pipeline completo.

\textbf{Baselines ($\geq$2).} Llama-8B, DeepSeek-R1-14B, DeepSeek-Coder-V2, Claude 3.5 Sonnet, GPT-4o (6 configuraciones incluyendo dual).

\textbf{Preguntas.} PRI-G1: ¿El dual supera a GPT-4o en tasa de compilación? PRI-G2: ¿El dual supera a solo Llama? PRI-V1--V3: verificación MSR/FNR/latencia (Parte II).

\section{Variables experimentales (1.3)}

\subsection{Variables independientes}
Configuración (6), modelo SLM/frontera, prompt de generación (\texttt{generator\_v1.txt}) / reparación (\texttt{generator\_repair\_v1.txt}), temperatura $T{=}0$, modo \texttt{VERIFIER\_MODE}.

\subsection{Variables dependientes}
\textbf{Generación:} \texttt{compile\_ok}, tasa de compilación (\%), \texttt{speedup\_proxy}, \texttt{gpu\_efficiency\_proxy}, latencia de generación (ms). \textbf{Verificación:} MSR, FNR, F1, JSON válido, latencia (ms).

\subsection{Variables controladas}
Benchmark 70 kernels + 20 tareas de generación; GPU Apple M4 24\,GB; seed 42; timeout 120\,s; mismas tareas para todas las configuraciones (within-subjects).

\section{Diseño experimental (1.4)}

\textbf{Tipo:} within-subjects. \textbf{Generación:} 20 tareas $\times$ 6 configs $\times$ 3 réplicas = 360 ejecuciones. \textbf{Verificación:} 70 kernels $\times$ 6 configs $\times$ 3 réplicas = 1260 evaluaciones. \textbf{Justificación:} McNemar/Wilcoxon pareados; control de variabilidad inter-tarea.

\section{Métricas de evaluación (1.5)}

\subsection{Tasa de compilación}
\textbf{Definición:} proporción de generaciones con \texttt{compile\_ok}. \textbf{Cálculo:} L1 AST + L2 léxico + L3 superficie Triton (@triton.jit, tl.load/store). \textbf{Interpretación:} éxito sintáctico/estructural pre-GPU. \textbf{Limitación M4:} no implica ejecución CUDA.

\subsection{Speedup proxy}
\textbf{Definición:} $s = t_{\mathrm{baseline}} / t_{\mathrm{pipeline}}$. \textbf{Cálculo:} $t_{\mathrm{baseline}}$ = tiempo checker en kernel de referencia; $t_{\mathrm{pipeline}} = t_{\mathrm{gen}} + t_{\mathrm{check}}$. \textbf{Limitación:} proxy operativo en M4, no speedup GPU real.

\subsection{GPU efficiency proxy}
\textbf{Definición:} compilaciones exitosas por segundo de generación. \textbf{Limitación:} no mide ocupación SM ni memoria GPU.

\subsection{MSR, FNR, F1, latencia, JSON válido}
Ver corrida de verificación; MSR = sensibilidad en \texttt{unsafe}; JSON válido = éxito operativo del verificador SLM.

\section{Tamaño del efecto esperado (1.6)}

Compilación: $\phi \approx 0.4$ (McNemar); $\Delta$compile $\geq 5$ pp (relevancia práctica). Verificación: $\phi \approx 0.4$ MSR; reducción FNR $\geq 8$ pp.

\section{Análisis de poder (1.7)}

$\alpha=0.05$, poder $1-\beta=0.80$. Generación: $N=20$ tareas. Verificación: 31 kernels \texttt{unsafe}.
